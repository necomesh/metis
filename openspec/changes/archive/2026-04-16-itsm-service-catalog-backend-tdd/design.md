## Context

`internal/app/itsm` 已经实现了服务目录分类、服务定义和种子数据的基础后端逻辑，但当前代码与既有 `openspec/specs/itsm-service-catalog/spec.md` 的契约并未完全对齐。探索结果显示，分类创建路径已经具备两层树限制与删除引用保护，但更新路径缺少等价校验；服务定义已具备部分编码唯一和经典工作流校验，但缺少 `engine_type` 列表过滤、分类存在性校验以及经典/智能字段互斥约束；catalog 与 service 的重复编码在数据库层虽然受唯一索引保护，但 HTTP 层未统一映射到 `409 Conflict`。此外，`internal/app/itsm` 当前几乎没有后端测试，导致这些行为无法被稳定回归验证。

该变更的约束是：

- 保持现有 ITSM 模块分层结构不变，继续沿用 handler -> service -> repo 的职责划分。
- 优先做最小修正，不引入新的产品能力或外部依赖。
- 测试需要适配当前仓库以 Go 后端单元/轻集成为主的风格，避免先搭建过重的全应用集成框架。

## Goals / Non-Goals

**Goals:**

- 让服务目录分类的创建、更新、删除在后端层面满足现有 spec 的结构约束和错误契约。
- 让服务定义的创建、更新、列表查询满足现有 spec 中已声明但尚未完全落地的过滤和引擎字段约束。
- 为 catalog/service/seed 路径补充可回归的后端测试，优先覆盖业务规则、HTTP 状态码映射和 seed 幂等性。
- 将数据库唯一索引错误收敛为稳定的领域错误或 HTTP `409`，避免调用方依赖数据库错误文本。

**Non-Goals:**

- 不新增新的前端页面、API 路由或数据模型字段。
- 不在本次变更中实现与 AI Agent、智能服务模板、前端服务目录浏览等无直接关联的未完成功能。
- 不引入完整端到端测试框架或改造整个 IOC/启动流程。

## Decisions

### 1. 在 service 层补齐 catalog 更新校验，并保持 handler 仅负责错误映射

`CatalogService.Create()` 已经承担父分类存在性与两层深度约束，因此 `Update()` 也应在 service 层复用相同的结构规则，而不是把校验留给 handler 或数据库。这样可以保证无论调用来自 HTTP 还是未来其他内部入口，分类树规则都由同一业务层控制。

本次设计将把以下校验放入 `CatalogService.Update()`：

- 更新 `parent_id` 时目标父分类必须存在
- 目标父分类必须是根分类，不能把节点挂到二级分类下
- 节点不能把自己设为自己的父分类
- 节点不能移动到自己的后代节点下

备选方案是仅在 handler 中根据 `parentId` 参数做校验，但这样会让 service 层对非法状态无感，也不利于单元测试，因此不采用。

### 2. 将重复编码冲突显式归一化为领域错误，再映射到 HTTP 409

目前 service 定义创建/更新已对重复 `code` 做部分预检查，但 catalog 创建/更新没有对应领域错误，且实际冲突仍可能直接从数据库唯一索引冒泡为通用错误。设计上应把“重复编码”视为稳定业务语义，而不是底层数据库实现细节。

本次设计采用两层保护：

- service 层优先做显式存在性检查，返回统一的领域错误
- handler 层将该领域错误映射为 `409 Conflict`

如果仍有数据库唯一约束穿透，service 层也应尽量识别并归一化，而不是把原始数据库错误直接暴露给 API。

备选方案是仅依赖预查询防冲突，但并发情况下仍可能出现竞争窗口，因此仍需考虑数据库错误归一化。

### 3. 服务定义继续采用“当前 engine_type + 更新载荷”的合并校验模型

`ServiceDefService.Update()` 当前已经根据 `existing.EngineType` 与 `updates["engine_type"]` 组合判断是否需要校验 `workflow_json`。本次保持这一模式，并将其扩展到更完整的引擎约束：

- classic 服务允许经典字段，拒绝智能字段
- smart 服务允许智能字段，拒绝经典字段
- 切换 `engine_type` 时按切换后的目标引擎校验本次载荷

这样可以避免在 handler 中复制复杂判断，并保持更新路径对部分字段 patch 的兼容性。

备选方案是强制 create/update 请求按 engine type 拆成两个 handler DTO，但这会扩大接口变更范围，不符合最小修改目标。

### 4. `engine_type` 过滤在 repository 层实现，handler 只负责解析查询参数

服务定义列表是典型查询能力，过滤条件属于 repo 查询逻辑。设计上在 `ServiceDefListParams` 中增加 `EngineType *string`，由 handler 从 query 解析，再由 repo 构造 SQL 条件。这样现有分页和其他过滤逻辑可以保持同一入口，测试也能分别覆盖 handler 参数解析与 repo/service 查询结果。

### 5. 测试采用“service 优先、handler 补契约、seed 锁幂等”的三层策略

本次测试不从全应用集成入手，而是采用更小、更稳定的测试面：

- service tests：覆盖 catalog/service 的核心业务规则
- handler tests：覆盖关键 HTTP 状态码与错误消息映射
- seed tests：覆盖 catalog seed 的首次创建、幂等重入和软删除后重建

底层存储采用轻量 SQLite 测试数据库，直接迁移 ITSM 相关模型，避免为测试引入完整 main.go 启动流程。这样能最大化回归价值，同时把测试复杂度控制在当前仓库能承受的范围内。

### 6. 本次变更把“测试覆盖”视为 capability 的一部分，而非纯实现细节

由于当前问题本质上是“spec 已存在但后端行为未被稳定落实”，单纯修代码不足以保证能力真正可维护。因此该变更把关键测试用例上升为 spec 的一部分：系统不仅要满足行为要求，还必须以自动化测试覆盖这些后端契约。这样后续回归时，OpenSpec 和代码验证目标才能一致。

## Risks / Trade-offs

- [更新校验增强后可能暴露已有脏数据或调用路径] → 先以 service 层校验阻止新增非法状态，若发现历史脏数据，再单独处理数据修复。
- [重复编码冲突识别依赖数据库错误特征] → 优先采用显式查询 + 数据库错误兜底归一化，避免只绑定某一种数据库错误文本。
- [handler 测试需要构造依赖，测试样板增加] → 将测试基建压缩为最小 router + test DB，避免引入全 IOC 容器。
- [新增测试后实现改动节奏会变慢] → 这是有意的权衡，用更高的回归信心换取后续 ITSM 演进稳定性。
- [引擎字段互斥规则可能与现有未完成前端交互存在偏差] → 以后端 spec 为准收敛规则，若前端尚未对齐，再在后续 UI change 中跟进。
